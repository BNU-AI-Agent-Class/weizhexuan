## 各功能详细说明

### 选项A：search 搜索工具

**新增工具：`search(keyword, path=".")`**

功能：
- 递归遍历指定目录下的所有文件
- 自动跳过 `.git`、`__pycache__`、二进制文件（.pyc、.exe、图片等）
- 逐行查找包含关键词的内容
- 返回格式：`文件路径:行号: 该行内容`
- 最多返回50条结果，避免上下文爆炸
- 处理编码错误、权限错误，不会因为读不了文件而崩溃

**验收示例**：
```
你：在 demo_project 里搜一下，哪个文件用到了 add_note？
→ [调用] search → 定位到 store.py（定义）和 cli.py（调用）
```

---

### 选项B：code_reviewer 代码审查员子agent

**新增工具：`code_reviewer(task)`**

子agent人设修改：
- 不再是通用的"子助手"，而是专门的"资深代码审查员"
- 系统提示明确了审查要点：边界条件（空输入、索引越界）、异常处理、逻辑错误、崩溃点
- 要求工作流程：先浏览目录 → 通读所有相关代码 → 不要只看一个文件就下结论 → 最后返回详细报告
- 子agent自己也有 `read_file` 和 `bash` 工具，可以独立读文件浏览目录
- 只报告问题，不修改代码

**报告格式**：每个bug列出文件位置、问题描述、触发条件、修复建议

**验收示例**：
```
你：派审查员去 demo_project 检查代码质量，报告潜在 bug
→ 审查员通读后返回：analyze.py 的 summary() 在无标签时会 IndexError
```

### 选项C：防崩加强版

修复了两个漏洞：

1. **未知工具名处理**：
   - 原来：`TOOLS[name]` 当name不存在时直接抛 `KeyError` 崩溃
   - 修复：派发前先检查 `name in TOOLS`，不存在就告诉AI"没有这个工具，可用的有：xxx"，让它纠正

2. **重试次数限制**：
   - 原来：`except` 里只有 `continue`，模型一直犯错就无限循环
   - 修复：增加 `consecutive_failures` 计数器，连续失败5次自动放弃当前任务
   - 计数器在工具执行成功后清零，不会误杀正常长任务

3. **额外加固**：
   - 区分不同错误类型（JSON解析错、未知工具、参数错、工具执行错），给AI不同的提示信息
   - 参数错误时自动显示工具的函数签名，帮助AI纠正参数

## Capstone 加分题：Bug修复说明

### Bug位置
`demo_project/notes_app/analyze.py` 的 `summary()` 函数第19行

### 问题原因
```python
# 原代码（有bug）
def summary():
    notes = all_notes()
    counts = tag_counts()
    top_tag = counts[0][0]  # BUG: counts为空列表时，counts[0] 抛 IndexError
    ...
```
当所有笔记都不带标签时，`tag_counts()` 返回空列表 `[]`，此时 `counts[0]` 会直接抛出 `IndexError: list index out of range`。

### 修复方案
```python
# 修复后
def summary():
    notes = all_notes()
    counts = tag_counts()
    lines = [f"共有 {len(notes)} 条笔记。"]
    
    if not counts:
        # 没有标签时友好提示
        lines.append("还没有任何标签。")
        lines.append("给笔记加上 #标签 后就能看到标签统计啦！")
    else:
        top_tag = counts[0][0]
        lines.append(f"最常用的标签是 #{top_tag}。")
        lines.append(f"一共用过 {len(counts)} 个不同标签。")
    
    return "\n".join(lines)
```

## 使用方法

1. 确保已安装依赖：`pip install python-dotenv openrouter`
2. 在 `build-your-agents-main/` 目录下创建 `.env` 文件，填入：
   ```
   OPENROUTER_API_KEY=你的API密钥
   ```
3. 运行对应版本：
   ```bash
   # 选项A：搜索工具版
   python hw2/c1_toolbox_with_search.py
   
   # 选项B：代码审查员版
   python hw2/c3_code_reviewer.py
   
   # 选项C：防崩加强版
   python hw2/c5_guard_robust.py
   
   # 最终整合版（推荐）
   python hw2/c6_final.py
   ```

## 遇到的坑与收获

1. **子agent的工具隔离**：一开始子agent只能用bash，没法read_file，导致审查员读不了代码。后来给子agent单独配了工具集才解决。
2. **重试计数器清零时机**：一开始计数器在任务完成才清零，导致正常长任务中间一次失败就计数，后来改成每次工具执行成功就清零。
3. **search结果截断**：一开始没限制结果数量，搜一个常见关键词返回几百行直接把上下文塞满，后来加了50条上限。
4. **二进制文件读取**：一开始没跳过二进制文件，读到.png/.pyc时抛UnicodeDecodeError，后来加了后缀名黑名单和try-except。

## mini Claude Code 像与不像的地方（加分题反思）

**最像真Claude Code的地方**：
- 派code_reviewer子agent去审查代码，主对话只拿到结论，上下文保持干净——这和真CC派Task工具探索代码库的模式一模一样
- search工具能快速定位代码位置，不用手动grep，开发体验很接近
- 危险命令前停下来问y/n，这种"人在环"的安全感很真实

**最不像的地方**：
- 工具太少了，真CC有Glob、Grep、Edit、WebSearch等几十个工具
- 没有真正的文件编辑工具（只能整文件读写，不能精确替换某一行）
- 上下文压缩只是简单按消息条数触发，真CC是按token数精确计算
- 没有错误恢复和自我修正的复杂机制，模型犯一次错需要用户或反馈告诉它，不会自己debug

**觉得还差的机制**：
1. **精确编辑工具**：类似真CC的Edit工具，能精确替换文件中的某一段，而不是只能整文件重写
2. **Glob文件名搜索**：按文件名模式查找文件，和search内容搜索配合
3. **LSP/代码理解**：能跳转到定义、找引用，真正理解代码结构而不只是字符串匹配
4. **多轮纠错能力**：工具执行出错时能自己分析错误原因并修正，而不是简单重试
5. **技能系统**：能加载特定领域的技能提示词
6. **任务中断与恢复**：用户能中途打断，暂停/恢复任务
